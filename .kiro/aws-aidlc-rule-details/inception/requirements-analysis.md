# 需求分析（自适应）

**承担角色**：产品负责人

**自适应阶段**：始终执行。详细程度根据问题复杂性调整。

**参见 [depth-levels.md](../common/depth-levels.md) 获取自适应深度解释**

## 前置条件
- 工作区检测必须完成
- 逆向工程必须完成（如果是棕地项目）

## 执行步骤

### 步骤1：加载逆向工程上下文（如果可用）

**如果是棕地项目**：
- 加载 `aidlc-docs/inception/reverse-engineering/architecture.md`
- 加载 `aidlc-docs/inception/reverse-engineering/component-inventory.md`
- 加载 `aidlc-docs/inception/reverse-engineering/technology-stack.md`
- 在分析请求时使用这些来理解现有系统

### 步骤2：分析用户请求（意图分析）

#### 2.1 请求清晰度
- **清晰**：具体、定义明确、可操作
- **模糊**：一般、模糊、需要澄清
- **不完整**：缺少关键信息

#### 2.2 请求类型
- **新功能**：添加新功能
- **错误修复**：修复现有问题
- **重构**：改进代码结构
- **升级**：更新依赖或框架
- **迁移**：迁移到不同技术
- **增强**：改进现有功能
- **新项目**：从头开始

#### 2.3 初始范围估算
- **单个文件**：对一个文件的更改
- **单个组件**：对一个组件/包的更改
- **多个组件**：跨多个组件的更改
- **系统级**：影响整个系统的更改
- **跨系统**：影响多个系统的更改

#### 2.4 初始复杂性估算
- **微不足道**：简单、直接的更改
- **简单**：清晰的实现路径
- **中等**：一些复杂性，多个考虑因素
- **复杂**：显著复杂性，许多考虑因素

### 步骤3：确定需求深度

**基于请求分析，确定深度：**

**最小深度** - 使用条件：
- 请求清晰简单
- 不需要详细需求
- 只记录基本理解

**标准深度** - 使用条件：
- 请求需要澄清
- 需要功能和非功能需求
- 正常复杂性

**全面深度** - 使用条件：
- 涉及多个利益相关者的复杂项目
- 高风险或关键系统
- 需要可追溯性的详细需求

### 步骤4：评估当前需求

分析用户提供的任何内容：
   - 意图陈述或描述（已在audit.md中记录）
   - 现有需求文档（如果提到则搜索工作区）
   - 粘贴的内容或文件引用
   - 将任何非markdown文档转换为markdown格式

### 步骤5：彻底完整性分析

**关键**：使用全面分析评估需求完整性。当有任何歧义或缺失细节时默认提问。

**必须**：评估所有这些领域，对任何不清楚的地方提问：
- **功能需求**：核心功能、用户交互、系统行为
- **非功能需求**：性能、安全、可扩展性、可用性
- **用户场景**：用例、用户旅程、边缘情况、错误场景
- **业务上下文**：目标、约束、成功标准、利益相关者需求
- **技术上下文**：集成点、数据需求、系统边界
- **质量属性**：可靠性、可维护性、可测试性、可访问性

**有疑问时，提出问题** - 不完整的需求导致糟糕的实现。

### 步骤6：生成澄清问题（主动方法）
   - **始终**创建 `aidlc-docs/inception/requirements/requirement-verification-questions.md`，除非需求异常清晰完整
   - 对任何缺失、不清楚或模糊的领域提问
   - 专注于功能需求、非功能需求、用户场景和业务上下文
   - 要求用户直接在问题文档中填写所有[Answer]:标签
   - 如果为答案提供多选选项：
     - 将选项标记为A、B、C、D等
     - 确保选项互斥且不重叠
     - **始终**包含自定义响应选项："X) 其他（请在下面的[Answer]:标签后描述）"
   - 等待用户在文档中回答
   - **必须**：分析所有答案的歧义，如需要创建后续问题
   - **必须**：继续提问直到所有歧义解决或用户明确要求继续

### 步骤7：生成需求文档
   - 创建 `aidlc-docs/inception/requirements/requirements.md`
   - 在顶部包含意图分析摘要：
     - 用户请求
     - 请求类型
     - 范围估算
     - 复杂性估算
   - 包含功能和非功能需求
   - 纳入用户对澄清问题的答案
   - 提供关键需求的简要摘要

### 步骤8：更新状态跟踪

更新 `aidlc-docs/aidlc-state.md`：

```markdown
## 阶段进度
### 🔵 启动阶段
- [x] 工作区检测
- [x] 逆向工程（如果适用）
- [x] 需求分析
```

### 步骤9：记录并继续
   - 在 `aidlc-docs/audit.md` 中记录批准提示和时间戳
   - 按此结构呈现完成消息：
     1. **完成公告**（必须）：始终以此开始：

```markdown
# 🔍 需求分析完成
```

     2. **AI摘要**（可选）：提供需求的结构化要点摘要
        - 格式："需求分析已识别[项目类型/复杂性]："
        - 列出关键功能需求（要点）
        - 列出关键非功能需求（要点）
        - 如果相关，提及架构考虑或技术决策
        - 不要包含工作流指令（"请审查"、"让我知道"、"进入下一阶段"、"在我们继续之前"）
        - 保持事实和内容导向
     3. **格式化工作流消息**（必须）：始终以此确切格式结束：

```markdown
> **📋 <u>**需要审查：**</u>**  
> 请检查需求文档：`aidlc-docs/inception/requirements/requirements.md`



> **🚀 <u>**下一步？**</u>**
>
> **您可以：**
>
> 🔧 **请求变更** - 如果基于您的审查需要修改需求，请要求修改
> [如果将跳过用户故事，添加此选项：]
> 📝 **添加用户故事** - 选择包含**用户故事**阶段（基于项目简单性当前跳过）  
> ✅ **批准并继续** - 批准需求并进入**[用户故事/工作流规划]**

---
```

**注意**：仅当将跳过用户故事阶段时包含"添加用户故事"选项。用实际下一阶段名称替换[用户故事/工作流规划]。

   - 在继续之前等待明确的用户批准
   - 记录批准响应和时间戳
   - 在aidlc-state.md中更新需求分析阶段完成
