# 过度自信预防指南

## 问题陈述

AI-DLC 表现出过度自信，没有提出足够的澄清问题，即使对于复杂的项目意图陈述也是如此。这导致做出假设而不是收集适当的需求。

## 根本原因分析

过度自信问题是由多个阶段中鼓励跳过问题的指令引起的：

1. **功能设计**："如果不适用则跳过整个类别"
2. **用户故事**："将类别用作灵感，而不是强制性检查清单"
3. **需求分析**：类似的模式鼓励最少的提问
4. **NFR需求**："仅当"条件阻止彻底分析

这些指令告诉AI避免提问，而不是鼓励全面的需求收集。

## 实施的解决方案

### 更新的问题生成理念

**旧方法**："仅在绝对必要时提问"
**新方法**："有疑问时，提出问题 - 过度自信导致糟糕的结果"

### 所做的关键更改

#### 1. 需求分析阶段
- 从"仅在需要时"改为"除非异常清晰，否则始终创建问题"
- 添加了全面的评估领域（功能、非功能、业务上下文、技术上下文）
- 强调主动提问方法

#### 2. 用户故事阶段
- 删除了"跳过整个类别"指令
- 添加了要评估的全面问题类别
- 增强了答案分析要求
- 加强了后续问题要求

#### 3. 功能设计阶段
- 用全面评估替换了"仅当"条件
- 添加了更多问题类别（数据流、集成点、错误处理）
- 加强了歧义检测和解决要求

#### 4. NFR需求阶段
- 扩展了基本NFR之外的问题类别
- 添加了可靠性、可维护性和可用性考虑
- 增强了技术歧义的答案分析

### 新的指导原则

1. **默认提问**：当有任何歧义时，提出澄清问题
2. **全面覆盖**：评估所有相关类别，不要跳过领域
3. **彻底分析**：仔细分析所有用户响应的歧义
4. **强制后续**：为任何不清楚的响应创建后续问题
5. **不带歧义继续**：在所有歧义解决之前不要继续

## 实施指导原则

### 问题生成
- 评估所有问题类别，不要跳过任何
- 在澄清可以提高质量的地方提问
- 在每个阶段包含全面的问题类别
- 默认包含而不是排除问题

### 答案分析
- 寻找模糊响应："取决于"、"也许"、"不确定"、"混合"、"介于之间"
- 检测未定义的术语和对外部概念的引用
- 识别矛盾或不完整的答案
- 为任何歧义创建后续问题

### 后续问题
- 当检测到歧义时创建单独的澄清文件
- 提出具体问题解决每个歧义
- 在所有不清楚的响应澄清之前不要继续
- 要彻底 - 过度澄清比澄清不足更好

## 质量保证

### 需要注意的红旗
- 复杂项目的阶段在不提任何问题的情况下完成
- 在模糊或歧义的用户响应下继续
- 在没有理由的情况下跳过整个问题类别
- 做假设而不是要求澄清

### 成功指标
- 为项目复杂性提出适当数量的澄清问题
- 彻底分析用户响应，需要时进行后续
- 在进入实施之前有清晰、无歧义的需求
- 由于前期澄清更好，减少了后期阶段的变更需求

## 维护

在以下情况下应参考此指南：
- 向AI-DLC添加新阶段
- 更新现有阶段指令
- 审查AI-DLC过度自信问题的性能
- 培训团队成员AI-DLC问题生成原则

## 关键要点

**前期提出太多问题比基于错误假设做出不正确的假设要好。** 前期提出澄清问题的成本远低于基于假设实施错误解决方案的成本。
